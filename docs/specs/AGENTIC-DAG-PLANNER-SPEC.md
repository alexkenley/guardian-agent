# Agentic DAG Planner Specification (Phase 1)

## Overview
This specification details the implementation of Phase 1 of the Agentic Uplift Plan: The Meta-Planner DAG. The goal is to evolve the `IntentGateway` and introduce a `TaskPlanner` and `AssistantOrchestrator` to handle complex, multi-step intents using a Directed Acyclic Graph (DAG) of sub-tasks.

## Core Components

1.  **Intent Gateway Updates (`src/runtime/intent-gateway.ts`)**
    *   **Classification:** Enhance the gateway to classify intents as either "simple" (direct routing) or "complex" (requiring planning).
    *   **Routing:** Route "complex" intents to the new `TaskPlanner`.

2.  **Task Planner (`src/runtime/planner/task-planner.ts`)**
    *   **Responsibility:** Given a complex objective, generate a DAG of actionable sub-tasks.
    *   **DAG Structure:** Each node in the DAG represents a specific action, tool call, or delegation to an existing skill (e.g., `automation-builder`, `coding-workspace`).
    *   **Dependencies:** Nodes must define dependencies (edges) to ensure correct execution order (e.g., Node B depends on the output of Node A).

3.  **Assistant Orchestrator (`src/runtime/planner/orchestrator.ts`)**
    *   **Responsibility:** Execute the DAG generated by the `TaskPlanner`.
    *   **Execution Engine:** Traverse the DAG, executing nodes whose dependencies are met.
    *   **State Management:** Track the status of each node (pending, running, success, failed).
    *   **Delegation:** When a node delegates to a skill (e.g., coding), the orchestrator pauses that branch of the DAG, invokes the skill via the standard `runTool` or skill execution pathway, and resumes when the skill completes.

## Data Models

```typescript
// Proposed DAG Node Structure
interface PlanNode {
  id: string;
  description: string;
  dependencies: string[]; // IDs of nodes that must complete first
  actionType: 'tool_call' | 'skill_delegation' | 'routine_execution';
  target: string; // Tool name, skill ID, or routine ID
  inputPrompt: string; // The specific instruction for this node
  status: 'pending' | 'running' | 'success' | 'failed';
  result?: unknown;
}

interface ExecutionPlan {
  id: string;
  originalObjective: string;
  nodes: Record<string, PlanNode>;
  status: 'planning' | 'executing' | 'completed' | 'failed';
}
```

## Security & Integration Constraints
*   **No Interference:** The `TaskPlanner` is a meta-planner. It must *not* attempt to perform the specialized logic of existing skills (e.g., writing code, managing the file system directly). It must delegate.
*   **Approvals:** The `AssistantOrchestrator` must respect the existing Guardian approval layers. If a node delegates to a mutating skill, that skill's tool calls still trigger standard pending actions.
*   **Memory:** The orchestrator must not bypass the Second Brain. Routine executions and memory reads must flow through established service boundaries.
