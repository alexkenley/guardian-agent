# Guardian Agent x D.U.H. x Multica Uplift Plan

**Status:** Draft
**Date:** 2026-04-16
**Source Research:** [duh-research/](../../duh-research), [multica-research/](../../multica-research)

## Executive Summary

This plan outlines technical uplifts for Guardian Agent's security and coding assistant layers, inspired by the architectures of `duh` and `multica`. The focus is on fine-grained taint tracking, proactive security scanning, and speculative execution.

## 1. Security Architecture (The "Defensive Suite")

### 1.1 Structured Taint Envelopes (Replacing Coarse Trust)
- **Concept:** Instead of trying to mutate JavaScript string primitives (which breaks native APIs), we will extend our existing `trustLevel` and `taintReasons` metadata to wrap specific arguments in structured "Taint Envelopes" within the `ToolExecutionRequest`.
- **Benefit:** When evaluating a shell command in Layer 2 (Guardian Agent), we will know exactly which arguments were generated by the model vs. provided by the user, making injection detection deterministic.
- **Steps:**
    - Update `ToolExecutor` and `ChatAgent` to pass structured taint metadata alongside string arguments.

### 1.2 Taint-Driven Mandatory Approvals
- **Concept:** Rather than building a redundant token system, we will uplift the existing `PendingActionStore`. High-risk tools (Bash, Write) will automatically trigger an `approval_required` block if they receive tainted input, even if the user's global policy is `autonomous`.
- **Benefit:** Prevents "stealth" command execution by forcing explicit user approval via our standard UI for any action identified as risky by the taint tracker.
- **Steps:**
    - Update `ToolExecutor` to check for tainted inputs on `dangerous` tool calls and route to `PendingActionStore`.

### 1.3 Lethal Trifecta Heuristic
- **Concept:** A session-start check that detects the "Lethal Trifecta": `READ_PRIVATE` + `READ_UNTRUSTED` + `NETWORK_EGRESS`.
- **Benefit:** Warns the user immediately if they are about to run a task that has the classic "exfiltration" capability profile.

### 1.4 Dynamic Security Scanners
- **Concept:** Integrate static environment and repo scanners (e.g., `pip-audit`, `detect-secrets`, `semgrep`).
- **Benefit:** Proactive detection of vulnerabilities in the workspace before the agent even starts.
- **Steps:** Add `SecurityScannerRegistry` to `src/runtime/security-scanners.ts`.

## 2. Coding Assistant Uplifts

### 2.1 Ghost Snapshots (Speculative FS Overlay)
- **Concept:** An in-memory filesystem overlay that intercepts writes during exploration.
- **Benefit:** The coding assistant can perform "trial" edits, read back the modified files, and even run tests (if the test runner supports the overlay) without actually touching the disk.
- **Steps:**
    - Create a `GhostOverlay` service in `src/runtime/fs-overlay.ts`.
    - Update `fs_read` and `fs_write` tools to check the overlay first.
    - Add `/ghost merge` and `/ghost discard` commands to the UI.

### 2.2 Progressive Execution HUD
- **Concept:** A real-time UI component showing discrete execution steps.
- **Benefit:** Improves transparency. "Step 1/5: Inspecting", "Step 2/5: Planning"...
- **Steps:** Add `TaskProgress` events to the `ToolExecutor`.

### 2.3 Rigid Plan-Mode State Machine
- **Concept:** Formalize `code_plan` into a mandatory workflow state.

## 3. Implementation Phasing

1.  **Phase 1 (Security Core):** Structured Taint Envelopes and Taint-Driven Mandatory Approvals.
2.  **Phase 2 (Speculative Execution):** `GhostOverlay` and `code_plan` state machine.
3.  **Phase 3 (Integrations):** Additional dynamic security scanners.
