# Agentic System Hardening and Production Wiring

## 1. Overview
The experimental phases of the Agentic DAG Uplift introduced the logic, data models, and prompt structures for proactive planning, semantic reflection, recovery, and compaction. 

However, `SECURITY.md` demands that all execution, LLM prompting, and memory mutations must be subject to Guardian Admission, `ToolExecutor` approval/sandboxing, and `OutputGuardian` scanning. The current mocked `executeNode` bypasses these invariants.

This document serves as the final implementation plan to hard-wire the agentic components into GuardianAgent's secure runtime.

## 2. Wiring Node Execution (The Executor Bridge)
The `AssistantOrchestrator` requires a real implementation for `executeNode`. This must be injected from `chat-agent.ts` and dispatch to the correct supervisor-owned runtime paths:

*   **`actionType: 'tool_call'`**: Must map directly to `this.tools.runTool()`. This ensures the action is subjected to the `ShellCommandController`, `SsrfController`, and `GuardianAgent` inline LLM review before execution. It also ensures the action honors `assistant.tools.policyMode` (e.g., `approve_each`).
*   **`actionType: 'execute_code'`**: The task planner generates a sandboxed script. This must not bypass isolation. The executor bridge must wrap the script payload and dispatch it as a `code_remote_exec` tool call, ensuring it runs within the configured remote sandbox or bounded process sandbox, subject to the same strict timeouts and resource limits as any other code execution.
*   **`actionType: 'skill_delegation'`**: Must be mapped to a recursive planner call or a structured skill invocation through `this.skillResolver`.
*   **`actionType: 'delegate_task'`**: Must be implemented by spawning a new `AgentContext` or dispatching a message back into the routing layer as a sub-agent with a constrained budget. For this initial production wiring, we will map this to the existing conversational orchestration pattern.

## 3. Securing Reflection & Compaction LLMs
`TaskPlanner`, `SemanticReflector`, `RecoveryPlanner`, and `ContextCompactor` all rely on a `chatFn`. 
*   **Action:** Ensure that the `chatFn` injected from `chat-agent.ts` is `ctx.llm!.chat`. `ctx.llm` is guaranteed by the framework to be a `GuardedLLMProvider`, meaning the outputs of these internal reasoning steps are automatically scanned for secrets, PII, and injection vectors by the `OutputGuardian` before the orchestrator consumes them.
*   **Tech Debt Fix:** All components currently use `JSON.parse()`. This is brittle. We must refactor them to use `import { parseStructuredJsonObject } from '../util/structured-json.js';` to ensure safe, resilient parsing of the LLM output.

## 4. Persisting the Learning Queue
The `ReflectiveLearningQueue` currently uses `console.log`.
*   **Action:** The queue must be wired to use the `AgentMemoryStore` (if available) or emit a structured event (`ctx.emit`) so the supervisor process can log the learned insight into the durable, trust-aware memory system. The insight should be tagged with provenance indicating it was autonomously generated by the Agentic DAG.

## 5. Implementation Steps
1.  **Refactor JSON Parsing:** Update all planner components to use `parseStructuredJsonObject`.
2.  **Wire the Executor Bridge:** In `chat-agent.ts`, implement the full `executeNode` function and pass it to the orchestrator.
3.  **Wire the Learning Queue:** In `chat-agent.ts`, implement the memory persistence callback for the learning queue.
4.  **Run Integration Tests:** Execute the full Vitest suite to guarantee no existing security boundaries were inadvertently disabled.
