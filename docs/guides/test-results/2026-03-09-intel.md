## Summary

This run validates real threat-intel behavior, not just relaxed assertions:

- The prerequisite probe now hits the actual registered read-only tool: `intel_summary`.
- Read-only tools (`intel_summary`, `intel_findings`) returned `succeeded`, not `failed`.
- Mutating tools (`intel_watch_add`, `intel_watch_remove`) executed successfully in `autonomous` mode.
- `intel_scan` produced a real finding ID, which was then used by `intel_draft_action`.
- Under `approve_by_policy`, mutating intel tools returned `pending_approval` and accepted denial decisions.
- Read-only and network intel tools remained allowed without approval under `approve_by_policy`.

## What Was Fixed

The changes were in the harness and guide, not the threat-intel runtime:

| Issue | Previous behavior | Fix |
|------|-------------------|-----|
| Wrong tool name | Harness tested `intel_watch_list`, which does not exist | Switched to `intel_summary` |
| False-positive pass criteria | `failed` counted as pass for probe/read-only tests | Tightened to require `success=true` or `status=succeeded` |
| Stale policy overrides | Global mode change could leave old per-tool overrides in effect | Harness now sets explicit per-tool intel policies before approval tests |
| Invalid draft action input | Harness used fake `findingId` and invalid action type | Harness now uses a real `findingId` from `intel_scan` and valid `draft_response` type |
| PowerShell compatibility | Used `??`, which is not safe for Windows PowerShell 5.1 | Replaced with helper fallback logic |
