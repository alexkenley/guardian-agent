# Memory System Guide

GuardianAgent's memory system provides persistent, searchable conversation history and per-agent knowledge bases. This guide covers architecture, configuration, and usage.

## Architecture

The memory system has four layers:

```
Layer 1: Knowledge Base (AgentMemoryStore)
  Per-agent markdown files (~/.guardianagent/memory/{agentId}.md)
  Always loaded into LLM context as <knowledge-base> block
  Written via memory_save tool or automatic memory flush

Layer 2: FTS5 Search Index (ConversationService)
  SQLite FTS5 virtual table with BM25 ranking
  Synced via triggers on INSERT/DELETE
  Queried via memory_search tool

Layer 3: Conversation History (ConversationService)
  SQLite-backed session storage with sliding window context
  Automatic trimming to maxContextChars with memory flush
  Session rotation, listing, and restore

Layer 4: Memory Flush (automatic)
  Detects when messages are dropped from context window
  Extracts key content and persists to knowledge base
  Ensures important facts survive context trimming
```

### Data Flow

```
User Message
    |
    v
[Knowledge Base loaded into system prompt]
    |
    v
[Conversation history trimmed to maxContextChars]
    |--- Messages that don't fit ---> [Memory Flush] ---> Knowledge Base
    |
    v
[LLM receives: system prompt + knowledge base + recent history + user message]
    |
    v
[Response recorded to conversation_messages + FTS5 index updated via trigger]
```

## Configuration

All settings live under `assistant.memory` in `~/.guardianagent/config.yaml`:

```yaml
assistant:
  memory:
    enabled: true
    maxTurns: 12              # Max user+assistant turns per session
    maxMessageChars: 4000     # Truncate individual messages beyond this
    maxContextChars: 12000    # Max chars from history included in LLM context
    retentionDays: 30         # Auto-delete conversation records older than this

    # Per-agent persistent knowledge base
    knowledgeBase:
      enabled: true
      basePath: ~/.guardianagent/memory    # Directory for per-agent .md files
      maxContextChars: 4000               # Max chars loaded into LLM context
      maxFileChars: 20000                 # Warning threshold for file size
      autoFlush: true                     # Auto-persist dropped context
```

## Memory Tools

Three tools are registered in the `memory` category:

### memory_search

Search conversation history using FTS5 full-text search with BM25 ranking.

```
Tool: memory_search
Risk: read_only
Category: memory
Parameters:
  query (required): Search query — words, phrases, FTS5 syntax
  limit (optional): Max results (default 10, max 50)
```

Supports FTS5 match expressions. Falls back to substring search if FTS5 is unavailable.

**Example queries:**
- `"project deadline"` — find messages mentioning project deadlines
- `TypeScript AND testing` — boolean search
- `"error 404"` — exact phrase match

### memory_get

Retrieve the persistent knowledge base for the current agent.

```
Tool: memory_get
Risk: read_only
Category: memory
Parameters:
  agentId (optional): Agent ID (defaults to current agent)
```

### memory_save

Save a fact, preference, or summary to the persistent knowledge base.

```
Tool: memory_save
Risk: mutating
Category: memory
Parameters:
  content (required): The fact or preference to remember
  category (optional): Heading for organization (e.g., "Preferences", "Facts")
```

**Example usage by the agent:**
- User says "remember that I prefer dark mode" -> agent calls `memory_save({ content: "User prefers dark mode", category: "Preferences" })`
- User says "my project uses PostgreSQL" -> agent calls `memory_save({ content: "Project database is PostgreSQL", category: "Project Notes" })`

## Knowledge Base File Format

Each agent's knowledge base is a markdown file organized by category:

```markdown
## Preferences
- User prefers dark mode _(2025-01-15)_
- Use TypeScript for all new code _(2025-01-17)_

## Facts
- User name is Alex _(2025-01-16)_
- Primary workspace is /home/alex/projects _(2025-01-18)_

## Context from 2025-01-20
- [user] What's the deployment process?
- [assistant] The deployment uses GitHub Actions with staging -> production flow...
```

The `## Context from YYYY-MM-DD` sections are auto-generated by the memory flush system.

## FTS5 Full-Text Search

The conversation database includes an FTS5 virtual table (`conversation_messages_fts`) that provides:

- **BM25 ranking**: Results scored by relevance, not just recency
- **Porter stemming**: "running" matches "run", "runs", etc.
- **Unicode support**: Handles international characters correctly
- **Content-sync**: No data duplication — FTS index references conversation_messages by rowid
- **Auto-sync triggers**: INSERT/DELETE on conversation_messages automatically updates the FTS index
- **Schema upgrade**: Existing databases get FTS rebuilt on first access

### Graceful Degradation

If FTS5 is not compiled into the SQLite build, the system falls back to:
- In-memory: case-insensitive substring matching
- SQLite without FTS5: substring search via `LIKE` (not currently implemented, returns empty)

Check availability: `conversationService.hasFTS` returns `true` when FTS5 is active.

## Memory Flush

When `buildMessages()` trims conversation history to fit `maxContextChars`, messages that don't fit are "flushed" to the knowledge base:

1. The sliding window walks backwards from the most recent message
2. When the character budget is exhausted, earlier messages are identified as "dropped"
3. If 2+ messages are being dropped, the `onMemoryFlush` callback fires
4. The callback extracts content previews (up to 200 chars per message, max 10 messages)
5. These are appended to the agent's knowledge base as a dated context block

### Flush behavior

- Only fires when substantive content is being dropped (2+ messages)
- Each message preview is capped at 200 characters
- Max 10 messages per flush event
- Flush failures are silently caught — never breaks message building
- Controlled by `knowledgeBase.autoFlush` config (default: `true`)

## Comparison with OpenClaw

| Feature | GuardianAgent | OpenClaw |
|---------|--------------|----------|
| **Storage backend** | SQLite with FTS5 + optional QMD hybrid search | SQLite FTS5 + optional QMD sidecar |
| **Search** | FTS5 BM25 + QMD hybrid (BM25 + vector + LLM re-rank) | Hybrid BM25 + vector similarity |
| **Knowledge base** | Per-agent markdown files | `MEMORY.md` + daily logs |
| **Memory flush** | Auto-persist dropped context | LLM-prompted flush before compaction |
| **Temporal decay** | No | Configurable half-life scoring |
| **MMR diversity** | No | Maximal Marginal Relevance |
| **Embedding providers** | QMD (local, via CLI subprocess) | OpenAI, Gemini, Voyage, Ollama, local |
| **Cross-channel identity** | IdentityService with aliases | Not documented |
| **Session management** | Rotate, restore, list | Implicit via daily log files |
| **Security** | Guardian admission on all tools | Not documented |

### Key Differences

**GuardianAgent advantages:**
- All memory tools pass through Guardian security (capability checks, secret scanning, rate limiting)
- Cross-channel identity unification
- Explicit session management with rotate/restore
- SQLite security hardening (permissions, integrity checks)

**OpenClaw advantages:**
- Vector similarity search (semantic matching beyond keywords)
- LLM-powered memory flush (summarization, not just extraction)
- Temporal decay scoring (recent memories ranked higher)
- MMR for result diversity
- Multiple embedding provider support
