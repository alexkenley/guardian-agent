Purpose:
- Keep one pinned upstream Daytona reference snapshot in-repo while we implement and review Guardian's Daytona remote sandbox support.
- Record the small set of details that matter most for Guardian's current integration so we do not have to rediscover them from the full upstream text each time.

Key alignment points for Guardian:
- Daytona Cloud configuration uses `apiKey`, optional `apiUrl`, and optional `target`. The upstream docs list `DAYTONA_API_URL` and document the default API URL as `https://app.daytona.io/api`.
- Sandbox creation supports `networkAllowList` and `networkBlockAll`. The upstream docs describe `networkAllowList` as a comma-separated list of IPv4 CIDR blocks, which matches Guardian's `allowedCidrs` model for Daytona profiles.
- If both `networkBlockAll` and `networkAllowList` are set, Daytona gives precedence to `networkBlockAll`. Guardian already maps deny-all isolation to `networkBlockAll` and only emits `networkAllowList` when CIDRs are configured.
- The TypeScript SDK documents the filesystem and process methods Guardian currently uses: `uploadFiles`, `downloadFile`, `setFilePermissions`, `executeCommand`, `getWorkDir`, `getUserHomeDir`, and sandbox deletion.
- The upstream docs confirm the sandbox is a full Linux environment with a Toolbox API for filesystem and process execution. That matches Guardian's use of Daytona as a bounded remote execution lane for staged repo commands rather than only inline code snippets.

Notable operator-facing detail:
- API keys are permission-scoped. For Guardian's current sandbox lifecycle, the Daytona key should include the sandbox permissions needed to create and delete sandboxes. The upstream API-key section explicitly documents sandbox write/delete scopes.

Current Guardian fit check:
- Guardian's Daytona provider shape is aligned with the documented SDK surface.
- Guardian's current UI terminology of API Key, optional API URL, optional Target, outbound CIDR allowlist, and sandbox enablement is aligned with the upstream model.
- Guardian intentionally uses `process.executeCommand(...)` plus staged filesystem upload instead of `code_run(...)`, because Guardian needs bounded repo execution, artifact retrieval, and command-level verification rather than snippet-only execution.

Follow-up worth keeping in mind:
- The operator docs should continue to mention that Daytona API keys are permission-scoped, because that is an easy setup miss even though the rest of the Daytona path is simpler than Vercel.
